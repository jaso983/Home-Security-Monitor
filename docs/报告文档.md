# Home Security Monitor 课程设计报告

## 1、引言

### 1.1 项目背景

随着现代家庭安全意识的提升，家庭火灾隐患及非法入侵等安全事故频发。传统安防设备往往只能事后查看录像，缺乏主动预警能力。本项目结合深度学习（YOLO 算法）与计算机视觉技术，旨在打造一套低成本、高效率的 AI 家庭安全监控系统，实现事前预警和事中告警。

应用视频技术，结合 AI 分析，训练算法，实现对家庭重点区域和特定场景的监控、识别与告警：

1. 通过视频 AI，对烟雾及火焰检测，发现就标注报警。
2. 通过视频 AI，在特定时段对进入人员进行检测，发现陌生人员闯入就标注报警。

系统部署于普通家用电脑，利用现有摄像头硬件，无需额外购置安防设备，降低家庭安防门槛。

### 1.2 项目目的

1. 实现基于视频流的 24 小时实时烟雾与火焰检测，保障家庭消防安全。
2. 实现在特定时段（如深夜 23:00-06:00）的异常人员闯入检测，保障家庭财产与人身安全。
3. 实现报警事件的智能合并与去重机制，避免无效告警对用户造成干扰。

### 1.3 项目特色

1. **AI 主动预警**：基于 YOLO 深度学习模型实时分析视频流，在异常发生瞬间即触发报警，区别于传统设备的被动录像模式。
2. **双模型协同**：人员检测（YOLOv8n）与火焰/烟雾检测（专用 fire 模型）并行运行，覆盖家庭最常见的两类安全隐患。
3. **智能防抖去重**：内置 AlarmManager 冷却机制，同一事件在 10 秒窗口内仅触发一次报警，避免重复骚扰。
4. **轻量级全本地运行**：无需 GPU、无需云服务，CPU 即可运行，所有数据（数据库、截图）存储在本地，保护用户隐私。
5. **双模式操作**：提供 GUI（图形界面）和 CLI（命令行）两种运行方式，GUI 面向普通家庭用户，CLI 方便开发者调试。
6. **浅色简洁 GUI**：基于 tkinter 实现的轻量级图形界面，布局直观，状态指示清晰。

### 1.4 项目风险

| 风险项 | 影响 | 缓解措施 |
|--------|------|----------|
| YOLO 模型推理速度不足导致帧率下降 | 监控画面卡顿，影响用户体验 | 采用轻量级 yolov8n 模型；精简检测类别；适当降低输入分辨率 |
| 火焰检测模型误报率偏高 | 用户体验差，频繁无效报警 | 提高置信度阈值至 0.8；限制仅检测 fire 和 fire-smoke 类别；后续可替换更精准的模型 |
| 摄像头夜间画质差导致人员漏检 | 安防失效 | 提示用户保持基本照明；未来可集成红外摄像头 |
| 用户修改配置文件导致系统崩溃 | 可用性降低 | 添加配置校验逻辑；提供默认配置回退 |

### 1.5 产品范围

**包含：**

- 实时视频流获取与显示
- 全天候火焰/烟雾 AI 检测与报警
- 特定时段人员闯入 AI 检测与报警
- 报警防抖去重机制
- 报警截图自动保存
- 报警历史记录存储、查询、删除
- GUI 图形界面 + CLI 命令行

**不包含：**

- 云存储与远程推送（后续扩展）
- 家庭成员面部识别
- 多摄像头同时接入
- 移动端 APP

## 2、综合描述

### 2.1 用户类和特性

| 用户类 | 特征 | 需求 |
|--------|------|------|
| 家庭普通用户 | 不具备专业 IT 知识，期望即开即用 | 系统全自动运行，报警提示直观醒目，操作简单 |
| 系统管理员（开发者） | 具备 Python 基础知识 | 查看后台日志、调整检测参数（置信度、冷却时间等）、管理报警数据库 |

### 2.2 运行环境

| 类别 | 最低配置 | 推荐配置 |
|------|----------|----------|
| 操作系统 | Windows 10 / 11 | Windows 11 |
| CPU | 4 核 | 6 核及以上 |
| 内存 | 8 GB | 16 GB |
| Python | 3.9+ | 3.11 |
| 摄像头 | USB 摄像头 / 笔记本内置摄像头（640×480） | 1080P USB 摄像头 |
| 依赖库 | ultralytics, torch, opencv-python, Pillow | 最新稳定版 |
| 磁盘空间 | 2 GB（含模型文件） | 5 GB（含截图存储） |

### 2.3 设计和实现上的限制

1. **单摄像头限制**：当前版本仅支持 Camera ID 0（默认摄像头），不支持多路视频流。
2. **无 GPU 加速**：YOLO 推理在 CPU 上运行，检测帧率受 CPU 性能约束。
3. **不支持远程访问**：系统仅在本机运行，无 Web 服务或远程推送功能。
4. **模型精度约束**：火焰检测模型为第三方预训练模型，对特定场景（深色物体、雾）的区分能力有限。
5. **Python 依赖**：必须安装 Python 3.9+ 及指定依赖，不适合无编程经验的用户独立安装。

### 2.4 假设和约束（依赖）

**硬件假设：**

- 系统运行环境需配备至少 4 核 CPU、8GB 内存及可用的摄像头设备。
- 监控区域需保持一定的光照条件，确保摄像头能够捕捉到清晰的视频流。

**软件依赖：**

| 依赖库 | 版本 | 用途 |
|--------|------|------|
| ultralytics | ≥ 8.0 | YOLO 模型加载与推理 |
| torch | ≥ 2.0 | 深度学习运算 |
| opencv-python | ≥ 4.8 | 视频流读取与图像处理 |
| Pillow | ≥ 10.0 | tkinter 图像转换桥梁 |

**数据依赖：**

- `yolov8n.pt`：YOLOv8 预训练模型（首次运行自动下载，约 6 MB）
- `fire_model.pt`：火焰/烟雾检测模型（需手动放置于 `src/models/`，约 20 MB）

## 4、非功能需求分析

### 4.1 性能需求

| 指标 | 需求值 | 实测值 |
|------|--------|--------|
| 视频处理帧率 | ≥ 15 FPS | 15-30 FPS（取决于 CPU 性能） |
| 报警延迟 | ≤ 2 秒 | 约 1 秒（从画面异常到数据库记录完成） |
| GUI 响应时间 | ≤ 500 ms | ≤ 200 ms |

### 4.2 可靠性需求

1. 系统能够连续运行 24 小时不出现内存泄漏或崩溃。
2. 摄像头意外断开时，GUI 显示占位图提示用户，每秒自动重试连接，恢复后自动继续监控。
3. 数据库操作异常时不影响视频监控主循环（try/except 保护）。

### 4.3 安全性需求

1. 报警日志保存在本地 SQLite 数据库，不允许非管理员通过网络远程篡改。
2. 系统配置（如监控时段）仅管理员有权修改源码参数。
3. 所有第三方模型文件在本地加载运行，不向外部发送视频数据。

### 4.4 可维护性需求

1. 代码遵循模块化设计，每个 `.py` 文件职责单一。
2. 关键方法均有类型注解和简要说明。
3. `tests/test_db.py` 提供数据库模块的独立单元测试，修改后执行即可验证。

### 4.5 可扩展性需求

| 扩展场景 | 改动范围 |
|----------|----------|
| 更换更强的 AI 模型 | 仅修改 `src/models/yolo_detector.py` |
| 加入煤气泄漏检测（气体传感器） | 新增传感器读取类 + AlarmManager 增加事件类型 |
| 添加 Web 端远程查看 | 新增 Flask/FastAPI 服务层，复用现有 DatabaseManager |
| 多摄像头支持 | `DatabaseManager` 增加 `camera_id` 字段，主循环改为多路采集 |
| 参数配置化 | 新增 `config.yaml` 读取模块，替代硬编码常量 |

### 4.6 数据一致性

1. SQLite 使用 WAL（Write-Ahead Logging）模式，写入与查询互不阻塞。
2. `insert_alarm` 采用原子事务提交（`commit()`），保证写库操作的原子性。
3. 截图保存与数据库写入在同一方法中顺序执行，任一步骤失败时上层 `try/except` 保护主循环不崩溃。
