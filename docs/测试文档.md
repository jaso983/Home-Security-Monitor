# Home Security Monitor 测试文档

## 1、引言

### 1.1 编写目的

本文档旨在规范 Home Security Monitor（家庭安防监控系统）的测试流程，确保系统各功能模块满足需求规格说明书中定义的功能性与非功能性要求。通过系统化的测试验证，发现并记录潜在缺陷，为开发团队提供修复依据，最终交付稳定可靠的 AI 家庭安防产品。

### 1.2 测试目标

1. **功能正确性**：验证实时视频流获取、火焰/烟雾检测、人员闯入检测、报警防抖合并、历史记录存储五大核心功能均能正确运行。
2. **性能达标**：验证视频处理帧率稳定在 15 FPS 以上，报警延迟不超过 2 秒。
3. **可靠性**：验证系统连续运行稳定性，摄像头断连恢复能力。
4. **数据完整性**：验证报警记录正确写入 SQLite 数据库，截图文件完整保存。
5. **UI 可用性**：验证 GUI 界面布局正确、按钮响应正常、报警状态指示准确。

## 2、测试用例设计

### 2.1 测试数据准备

| 数据类型 | 内容 | 用途 |
|----------|------|------|
| 视频源 | USB 摄像头实时画面（640×480） | 基本功能测试 |
| 火焰样本 | 打火机火焰、纸片燃烧 | 火焰检测测试 |
| 烟雾样本 | 线香烟雾 | 烟雾检测测试 |
| 人员样本 | 测试人员在摄像头前走动 | 人员检测测试 |
| 无异常画面 | 空房间静态画面 | 误报抑制测试 |
| 数据库 | 自动生成 `src/core/alarms.db` | 数据库 CRUD 测试 |

### 2.2 测试策略

1. **单元测试**：对 DatabaseManager 模块执行独立 CRUD 测试（`tests/test_db.py`），验证建表、插入、查询、删除、清空功能。
2. **集成测试**：启动 GUI 应用，验证 YoloDetector → AlarmManager → DatabaseManager 的完整报警链路。
3. **系统测试**：模拟真实场景（火焰、闯入），确认端到端流程正常。
4. **回归测试**：每次代码修改后重新执行单元测试，确保已有功能不被破坏。

### 2.3 测试原则

1. 先单元后集成，先 CLI 后 GUI。
2. 每个测试用例必须可重复执行，结果一致。
3. 边界条件优先：空数据、极限值、异常输入。
4. 发现的缺陷需记录复现步骤、预期结果、实际结果。

## 4、结论

| 评估维度 | 结论 |
|----------|------|
| 数据库 CRUD | 全部通过，功能正常 |
| 火焰/烟雾检测 | 功能可用，置信度 0.8 下误报率明显降低，仅检测 fire 和 fire-smoke 类别，排除 fog/sol/factory-smoke |
| 人员检测 | 功能可用，置信度 0.8 + 仅 class=person |
| 报警防抖 | 10 秒冷却期正常工作 |
| 报警全链路 | 检测 → 截图 → 写库 → GUI 更新 全链路通 |
| GUI 界面 | 浅色主题可用，按钮响应正常，布局符合预期 |
| 性能 | 帧率稳定在 15-30 FPS 范围内 |

**整体评价**：系统各核心功能模块均已通过测试，满足需求文档中定义的功能点要求，可交付使用。

**遗留问题**：火焰模型对特定背景（深色物体、蓝色纺织品）在低置信度下仍存在偶发误报，当前通过提高置信度阈值至 0.8 + 限制检测类别缓解。后续可考虑替换更精准的火焰检测模型。

### 5.2 已知限制

1. 火焰检测模型 `fire_model.pt` 为第三方预训练模型，对非火焰场景（雾、工业烟雾、深色物体）的区分能力有限。
2. 人员检测仅限 `person` 类别（COCO class 0），不区分家庭成员与陌生人。
3. 系统依赖摄像头硬件，无摄像头时仅显示占位图，不具备远程视频源接入能力。
